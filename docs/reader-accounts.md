# Reader accounts

Two questions about the reader-login work (`packages/reader`, `@atproto/oauth-client-browser`, tracked alongside SLIMS-66): how a reader's session behaves once they've logged in on one Hedgerow site and visit another, and how far Hedgerow can push account creation itself before it's running its own identity infrastructure. Both come down to the same primitive — atproto OAuth sessions are scoped to a **client**, and a client is identified by a URL its metadata lives at, evaluated against a device session on the user's PDS.

## 1. Cross-site reader sessions

**The reality.** `BrowserOAuthClient` persists sessions in the browser's IndexedDB ([`oauth-client-browser` README](https://github.com/bluesky-social/atproto/blob/main/packages/oauth/oauth-client-browser/README.md): "the new tokens will be saved in the session store (in the browser's indexed DB)"). IndexedDB, like all standard web storage, is partitioned per top-level origin — a store written on `alice.example` is invisible to `bob.example`. There is no shared-storage mechanism across origins by design; that's the same boundary that makes cookies, localStorage, and service workers origin-scoped. So a reader who authorizes commenting on 50 independent Hedgerow-built blogs ends up with 50 separate OAuth grants, 50 separate DPoP keypairs, 50 separate IndexedDB stores — one per origin, full stop, regardless of whether they're all "the same" reader on the same PDS account.

This isn't only a storage-partitioning problem. Each Hedgerow site's OAuth client is registered under its own `client_id`, which per the atproto client-metadata model is the URL the metadata document is hosted at (`oauth-client-browser` README, "Client Metadata" section). The authorization server (the reader's PDS or entryway) tracks per-account, per-*client* authorization state — confirmed in [`oauth-provider`'s `account-manager.ts`](https://github.com/bluesky-social/atproto/blob/main/packages/oauth/oauth-provider/src/account/account-manager.ts) (`setAuthorizedClient`, `AuthorizedClientData`) and consumed in `checkConsentRequired` ([`oauth-provider.ts`](https://github.com/bluesky-social/atproto/blob/main/packages/oauth/oauth-provider/src/oauth-provider.ts)). A distinct `client_id` per blog means a distinct authorization record per blog even before storage partitioning enters the picture.

**What softens it in v1 — and one assumption that doesn't hold.** Re-auth on a new blog is a top-level redirect to the reader's PDS/entryway and back, which *is* exempt from storage partitioning (a full-page navigation is first-party wherever it lands), so the PDS's own login cookie carries over — the reader won't have to re-enter a password. What does **not** carry over, and this is worth being precise about: silent, no-click re-authorization. `request-manager.ts` forces `prompt: 'consent'` on every authorization request from a client that is `!isTrusted && !isFirstParty && token_endpoint_auth_method === 'none'` — i.e. exactly a plain browser SPA client, which `token_endpoint_auth_method: "none"` in the `oauth-client-browser` README's own example metadata confirms Hedgerow's reader client will be. The same code path outright rejects `prompt: 'none'` for such clients ("Public clients are not allowed to use silent-sign-on"). So the honest v1 story is: **login is silent, consent is not** — every distinct client_id gets an explicit "Authorize?" click on the PDS, every time, with no server-side memory that can be leaned on to skip it. That's a real UX cost per new blog, and it does not diminish with repeat visits unless the client stops being a bare public SPA (see below) or the PDS operator marks it `isTrusted`/`isFirstParty` — an allowlist only the PDS operator controls, and only for clients it recognizes.

**Options for actually solving this**, roughly in order of protocol-cleanliness:

- **(a) Shared auth broker at a neutral domain.** A `auth.hedgerow.social`-style service holds the OAuth session (ideally as a confidential/backend client — see below) and each blog embeds it via iframe/postMessage to ask "is this reader logged in?" This is the classic SSO-broker pattern, but it collides with exactly the storage partitioning above: a broker's storage, when the broker runs framed inside another origin, is third-party storage from the browser's point of view, and Safari's ITP and Firefox's ETP already block silent third-party storage access; Chrome requires `document.requestStorageAccess()`, which needs a user gesture and is not silent either. A broker *does* still help, but only by centralizing the "PDS remembers this device" cookie and, critically, by being a **backend service** rather than a browser SPA — a confidential client (its own keyset, `token_endpoint_auth_method` other than `none`) escapes the forced-consent branch entirely and is the only configuration under which `prompt: 'none'` is even legal per the provider code above. That's a strong argument for a BFF-style broker over a client-side one, and it's exactly what the `oauth-client-browser` README recommends generally ("it is recommended to use `@atproto/oauth-client-node`... more secure... longer lived tokens"). The tradeoff: Hedgerow becomes a hosted identity intermediary for every reader on every Hedgerow site, with the operational and trust burden that implies.
- **(b) FedCM.** AT Protocol's own proposal ([atproto.com/blog/working-to-decentralize-fedcm](https://atproto.com/blog/working-to-decentralize-fedcm)) is to make a PDS a FedCM Identity Provider and Hedgerow sites Relying Parties: instead of a redirect, the RP asks the *browser* "can you get me a credential for an atproto account?" and the browser mediates entirely, no top-level navigation, no per-origin storage needed on the RP side. Status as of mid-2026: the specific mechanism this needs — open, unregistered **Identity Provider Registration** (any PDS can act as an IdP without the RP pre-listing it) — is Stage 1/incubation in the FedID Community Group, not yet on a working-group track; Bluesky funded Emelia Smith with a $39k/12-month grant to push it there, next FedID meeting March 10 2026. Baseline FedCM itself (the RP/IdP-allowlist version, not the open-registration extension atproto needs) has full support in Chrome (mandatory for Google sign-in since August 2025) and is actively being implemented in Firefox (accounts pull, active mode, continuation API, login status API — shipping window 2025–2026). **Safari has no FedCM implementation planned at all** — Apple's stated authentication direction is passkeys, not FedCM — which means iOS/Safari readers get zero benefit from this path for the foreseeable future, a real gap given how much blog traffic is mobile Safari. The atproto team's own writeup names unresolved problems: silent-registration abuse, credential revocation, IdP-registration misuse, and a chicken-and-egg bootstrap (no RPs or IdPs are on this path yet). Read as: directionally the right long-term answer, not shippable in 2026, and permanently partial without Safari.
- **(c) Do nothing beyond per-origin polish.** Ship the honest v1 — persistent per-origin sessions, silent login via the PDS cookie, one explicit consent click per new site — and invest in making that click cheap (clear copy, fast redirect, DPoP refresh that never re-prompts once granted).

**The ask for the standard.site conversation:** this is not a Hedgerow-specific problem — every implementer of `site.standard.*` who wants live, in-place reader interaction hits the identical wall, because it's a browser storage-partitioning problem plus an atproto-OAuth-provider consent policy, not an application bug. Each project shipping its own broker means N slightly-different trust intermediaries for the same 50-blogs-one-reader problem. A shared, protocol-level answer — concretely, a FedCM profile for atproto that the working group can point every `site.standard` consumer at — beats that. Worth raising as a joint ask given Bluesky is already funding the FedCM work; a standard.site-aligned voice adds weight to getting open IdP registration onto the WG track sooner.

## 2. North star: white-label atproto signup from a blog

The vision: a reader with no prior atproto account clicks "join the conversation" on a Hedgerow-built blog, creates an account without leaving the page, and that account works identically on every other Hedgerow site and inside the official Bluesky app. No new protocol required — this is buildable today, verified against the `@atproto/pds` v0.4.226 source vendored in this repo's lockfile.

**Accounts live on a PDS, not on "Bluesky."** `com.atproto.server.createAccount` is a plain XRPC call any PDS implements ([`createAccount.d.ts`](https://github.com/bluesky-social/atproto/blob/main/packages/api/src/client/types/com/atproto/server/createAccount.ts), vendored at `node_modules/@atproto/api/.../createAccount.d.ts`); it takes `handle`, `email`, `password`, optional `inviteCode`, and returns a working session (`accessJwt`/`refreshJwt`) plus a `did`. The Bluesky app itself lets a user point at any custom PDS host at login, so an account minted on Hedgerow's own PDS is a first-class atproto identity there too — nothing Bluesky-specific about it.

Reading the PDS handler ([`createAccount.js`](https://github.com/bluesky-social/atproto/blob/main/packages/pds/src/api/com/atproto/server/createAccount.ts)) settles a few specifics worth having verified rather than assumed:

- **Invite codes are opt-in, off by default.** Gated by `ctx.cfg.invites.required`, driven by the `PDS_INVITE_REQUIRED` env var (`config/env.js`) — unset means open signup. Bluesky itself dropped invite-only in Feb 2024 and bsky.social signup is open today.
- **No CAPTCHA in the OSS PDS.** Nothing in `createAccount`'s handler or the reference `@atproto/pds` package touches CAPTCHA/proof-of-work. The only built-in anti-abuse is a coarse XRPC rate limit — 100 requests per 5-minute window on this endpoint (`rateLimit: { durationMs: 5 * MINUTE, points: 100 }`) — and email validation (`isEmailValid` + `disposable-email-domains-js` rejects throwaway addresses). Real abuse resistance (CAPTCHA, stricter per-IP limits) is something a self-hoster has to bolt on themselves; it isn't in the box.
- **XRPC is CORS-open by default.** The PDS mounts `cors()` with no origin allowlist (`app.use(cors({ maxAge: DAY / SECOND }))` in `dist/index.js`), so a blog's own JS can call `createAccount` directly, cross-origin, with no proxy needed.
- **Handles are configurable per instance.** `PDS_SERVICE_HANDLE_DOMAINS` (defaults to `.<hostname>`) is exactly the knob that gives a reader `alice.hedgerow.social` on signup. Custom-domain handle verification (a reader's own domain, later) is separate and already how Hedgerow authors' own handles presumably work.

**What Hedgerow would have to run.** The `bluesky-social/pds` distribution ([bluesky-social/pds](https://github.com/bluesky-social/pds)) is the deployable wrapper around `@atproto/pds` — install script, Docker Compose, Caddy for TLS. Its own sizing guidance: **1 CPU core, 1 GB RAM, 20 GB SSD supports 1–20 users** per instance — a genuinely small footprint for a handful of accounts, but a real fleet-management problem at any scale (sharding by instance, or accepting a much bigger single box past ~20 users, which the docs don't characterize). Two duties that are easy to underweight:
  - **Email is mandatory infrastructure, not optional.** `createAccount` requires a valid email and the PDS needs outbound SMTP to verify it (self-host or a transactional provider like Resend/SendGrid) — this is a dependency the write-only publish side never needed.
  - **Moderation exposure is real even though Bluesky's mod network absorbs report handling.** The PDS ships pointed at `mod.bsky.app` (`PDS_REPORT_SERVICE_URL`/`_DID`) for reports, but Hedgerow, as the account host, still carries the underlying legal and abuse-response duties of hosting other people's repos — this is qualitatively different from publishing your own posts.
  - **Backups are the operator's problem.** The distribution documents *where* data lives (`/pds`) but not a backup strategy; that's DIY.

**Portability is the anti-lock-in story, and it's real, not marketing.** An atproto account's durable identity is its DID (typically `did:plc`), not its handle or its host. A reader who signed up via Hedgerow's PDS can migrate their repo, blobs, and DID to any other PDS at will (`com.atproto.server` migration flows) — Hedgerow never becomes a walled garden the way a typical "sign up with us" flow would. Worth saying explicitly in outreach: creating an account here is not creating a Hedgerow account that happens to work elsewhere; it's creating a portable atproto account that happens to be easiest to get on Hedgerow.

**The embedded UX**, chaining what's now confirmed: signup form in the blog's comment box → `createAccount` XRPC call direct to Hedgerow's PDS (CORS-open, no redirect) → immediately into an OAuth grant for *that blog*. The grant step still involves a redirect to the PDS's own authorize page (XRPC alone can't mint a DPoP-bound OAuth session — `packages/reader` depends on `@atproto/oauth-client-browser`, so reader auth is OAuth, not the raw createAccount session token), but because Hedgerow controls the PDS on this path, it can mark its own reader client(s) `isTrusted`/`isFirstParty` server-side — the one legitimate way to escape the forced-consent branch from Section 1. That's a concrete, compounding payoff of self-hosting: it doesn't just enable white-label signup, it's the only lever Hedgerow has to make its *own* reader OAuth flow closer to silent, for accounts on its own infrastructure. It buys nothing for readers who already have a bsky.social or other third-party account — Hedgerow can't mark itself trusted on a PDS it doesn't run.

**Sequencing.**
- **v1 (ships now, zero infra):** no account → OAuth with `prompt: "create"`. Verified in the vendored source: `prompt` is a supported enum value (`oauth-types` `OAuthAuthorizationRequestParameters`), `AuthorizeOptions` passes it through `signIn()`, and — decisively — the provider's forced-consent branch explicitly exempts `prompt === 'create'` (`request-manager.ts`: "force consent for unauthenticated third party clients, **unless they are trying to create accounts**"). So the blog's "New here?" button calls `signIn('https://bsky.social', { prompt: 'create' })` and the reader signs up on Bluesky's own entryway *mid-OAuth-flow*, returning to the blog already authorized — no bsky.app detour, no manual "now come back and log in" step, still zero infrastructure. This is strictly better than the redirect-to-bsky.app plan and costs one option.
- **v2 (gated on real usage):** hosted-PDS white-label signup, in-page. Decision triggers worth setting now rather than deciding in the moment: sustained reader-signup volume that makes the redirect-to-bsky.app drop-off visibly costly; confidence in either building or contracting the SMTP/abuse/moderation operational surface above; and — separately — a real answer to whether Hedgerow wants to be an identity host at all, independent of the technical lift.

**Risks, plainly.** Running identity infrastructure as a two-person startup is a different risk class than running a static-site publisher: it means owning account recovery, being a target for credential-stuffing and spam-account creation (no CAPTCHA out of the box, remember), and inheriting the moderation and legal exposure of hosting other people's data even with Bluesky's mod network doing report triage. A search for existing "PDS hosting as a service, white-labeled for a third-party app" providers turned up self-hosting guides and a couple of one-click deploy templates (Railway, etc.) but no established managed offering built for this specific pattern — **this is a gap I couldn't fill with a citation, flagged rather than guessed at.** If one exists or emerges, partnering to offload the SMTP/abuse/backup burden while keeping the white-label front-end almost certainly beats self-hosting for a startup this size; worth an explicit look before committing to v2.
